1. 概览：为什么需要策略网络？

在机器人模仿学习 (Imitation Learning) 中，策略网络 (Policy) 的任务是：

观测 (图像 + 关节状态)  →  策略网络  →  动作 (关节目标角)

本项目中使用了两种策略网络：

  • ACT Action Chunking Transformer — 用于 SO-101 机械臂的推物任务
  • Diffusion Diffusion Policy — 用于 PushT 基准任务验证

2. ACT — Action Chunking with Transformers SO-101

2.1 核心思想

Action Chunking（动作分块）：不是每步只预测一个动作，而是一次性预测 chunk_size=100 步的未来动作序列，形成一个动作轨迹块。

这样做的好处：

  • 减少高频决策误差的累积
  • 天然编码了时序连贯性（动作之间平滑过渡）
  • 推理效率高：每 n_action_steps=100 步才调用一次模型

2.2 整体架构

┌──────────────────────────────────────────────────────────────────────────┐ │ ACT Architecture (Inference) │ │ │ │ Observation Input │ │ ┌──────────────┐ ┌──────────────┐ │ │ │ Image (B,3, │ │ State (B,6) │ │ │ │ 480, 640) │ │ (qpos) │ │ │ └──────┬───────┘ └──────┬───────┘ │ │ │ │ │ │ ▼ │ │ │ ┌──────────────┐ │ │ │ │ ResNet18 │ │ │ │ │ (ImageNet │ │ │ │ │ Pretrained) │ │ │ │ │ + 1×1 Conv │ │ │ │ └──────┬───────┘ │ │ │ │ (H×W tokens) │ (1 token) │ │ ▼ ▼ │ │ ┌────────────────────────────────────────┐ │ │ │ Transformer Encoder (4 layers) │ │ │ │ + Sinusoidal 2D Position Embedding │ │ │ │ Tokens: [latent | state | img_feat] │ │ │ └──────────────────┬─────────────────────┘ │ │ │ │ │ ▼ │ │ ┌────────────────────────────────────────┐ │ │ │ Transformer Decoder (1 layer) │ │ │ │ + Learnable Position Embedding │ │ │ │ Queries: 100 learnable tokens │ │ │ └──────────────────┬─────────────────────┘ │ │ │ │ │ ▼ │ │ ┌──────────────┐ │ │ │ Action Head │ → (B, 100, 6) # 100步动作序列，6个关节角 │ │ │ (Linear) │ │ │ └──────────────┘ │ └──────────────────────────────────────────────────────────────────────────┘

2.3 VAE 训练机制（关键创新）

ACT 在训练时引入了 VAE (Variational Autoencoder) 来处理人类演示中的多模态性——同一个目标可以通过不同路径达成。

Training 时额外的 VAE Encoder 分支： ┌────────────────────────────────────────────────────────────────┐ │ VAE Encoder (Training Only) │ │ │ │ [CLS token] + [robot_state] + [action_sequence (100 steps)] │ │ │ │ │ ▼ │ │ Transformer Encoder (4 layers) │ │ │ │ │ ▼ │ │ μ (mean) + σ (log variance) → z = μ + σ·ε (reparameterization) │ │ │ │ z 注入 Transformer Encoder 的第一个 token │ │ Inference 时 z = 0（不需要 VAE encoder） │ └────────────────────────────────────────────────────────────────┘

训练损失：

Loss = L1(actions, actionŝ) + kl_weight × DKL(q(z|x) ‖ N(0,1))

其中 kl_weight = 10，L1 loss 对动作序列做重建，KL loss 正则化潜在空间。

2.4 VAE 推理采样修复（重要）

问题发现 (2026-05-13)：LeRobot v0.5.1 中 ACT 推理时 VAE latent z 被设为全零向量，而非从先验分布 N(0,1) 采样。导致每个 episode 输出的动作序列完全相同，失去多样性。

VAE Latent z 在不同阶段的行为：

训练时 z = μ + σ · ε 重参数化采样，每次不同
推理时 (原版) z = [0, 0, 0, ..., 0] 全零！每次相同
推理时 (修复后) z = randn(32) ~ N(0,1) 从先验采样，每次不同

修复：将 modeling_act.py:454 的 torch.zeros 改为 torch.randn。

修复前后 eval 对比 (v9_r2, step 20K)

指标修复前 (zeros)修复后 (randn)
唯一轨迹数2 (ep1 + ep2-5 重复)5 (全部不同)
Avg min dist0.0794m0.0420m
推到 <5mm 次数1/53/5
Ep1 vs Ep2 action diff0.0000000.088534

关键结论：VAE latent z 是一个 32 维随机向量，决定了"怎么推"（推的轨迹、速度、风格）。就像人推东西——每次手法不完全一样，但都能把东西推到目标位置。此修复仅影响本地推理代码，不影响 RTX 上的训练过程（训练路径走的是第 449 行重参数化采样，未被修改）。

2.5 Action Queue 推理机制

推理时的 Action Queue 机制：

Inference Flow: Step 1: predict 100 actions → queue: [a1, a2, ..., a100] Step 2: execute a1 → queue: [a2, a3, ..., a100] Step 3: execute a2 → queue: [a3, ..., a100] ... Step 100: execute a100 → queue empty Step 101: predict 100 new actions → queue: [a1', a2', ..., a100']

每次 select_action() 从队列头部取一个动作，队列空了才重新推理。这使得模型推理频率为环境步频的 1/100。

2.6 为什么 chunk_size=100 会导致 O(n²) 显存？

关键细节： ACT 的 Transformer Decoder 中，queries 数量 = chunk_size。Self-Attention 的计算量和显存占用是 O(n²)（n = sequence length）。当 chunk_size=100 时 decoder 的 attention 矩阵为 100×100，是可接受的。但如果增大到 500，就是 500×500，显存急剧上升。这也是为什么 batch_size=128 已经是 RTX 6000D 85GB 的上限。

3. Diffusion Policy — 去噪扩散策略 PushT

3.1 核心思想

Diffusion Policy 将动作预测建模为一个去噪过程：

  • 训练时：对真实动作序列逐步加噪，学习如何还原
  • 推理时：从纯噪声开始，通过 100 步迭代去噪，生成动作序列

与 ACT 的一次性预测不同，Diffusion 通过多步迭代精炼动作，天然适合多模态分布。

3.2 整体架构

┌────────────────────────────────────────────────────────────────────────┐ │ Diffusion Policy Architecture │ │ │ │ Observation Input │ │ ┌──────────────┐ ┌──────────────┐ │ │ │ Image (B,3, │ │ State (B,2) │ │ │ │ 480, 640) │ │ (agent pos) │ │ │ └──────┬───────┘ └──────┬───────┘ │ │ │ │ │ │ ▼ │ │ │ ┌──────────────┐ │ │ │ │ ResNet18 │ │ (GroupNorm, not BatchNorm) │ │ │ + Spatial │ │ │ │ │ Softmax │ │ │ │ │ (32 keypoints│ │ │ │ │ → feature) │ │ │ │ └──────┬───────┘ │ │ │ │ │ │ │ └────────┬─────────┘ │ │ │ concat → global_condition │ │ ▼ │ │ ┌──────────────────────────────────────────────┐ │ │ │ Conditional U-Net 1D │ │ │ │ │ │ │ │ Input: noisy_action (B, horizon=16, dim) │ │ │ │ Condition: timestep + global_condition │ │ │ │ via FiLM modulation │ │ │ │ │ │ │ │ ┌────────┐ ┌────────┐ ┌────────┐ │ │ │ │ │ Down │→ │ Down │→ │ Down │ │ │ │ │ │ 512 │ │ 1024 │ │ 2048 │ │ │ │ │ └───┬────┘ └───┬────┘ └───┬────┘ │ │ │ │ │ │ │ │ │ │ │ ┌───┴────┐ ┌───┴────┐ ┌───┴────┐ │ │ │ │ │ Up │← │ Up │← │ Up │ │ │ │ │ │ 2048 │ │ 1024 │ │ 512 │ │ │ │ │ └────────┘ └────────┘ └────────┘ │ │ │ │ │ │ │ │ Output: predicted noise ε (B, 16, action_dim)│ │ │ └──────────────────────────────────────────────┘ │ │ │ │ │ ▼ │ │ DDPM Scheduler → denoise 100 steps → clean action │ │ │ │ │ ▼ │ │ Action (B, n_action_steps=8, action_dim) │ └────────────────────────────────────────────────────────────────────────┘

3.3 扩散过程详解

前向扩散（训练时加噪）

xt = √(ᾱt) · x0 + √(1 - ᾱt) · ε   其中 ε ~ N(0, I)

逐步往真实动作 x0 中添加高斯噪声，经过 num_train_timesteps=100 步后变成纯噪声。

反向去噪（训练目标）

Loss = ||ε - εθ(xt, t, obs)||²

U-Net 学习预测被添加的噪声 ε。知道噪声就能还原原始信号。

推理去噪（生成动作）

Inference: iterative denoising (100 steps) x100 ~ N(0, I) ← 纯噪声 │ ▼ U-Net + DDPM step x99 → x98 → ... → x1 → x0 = clean action sequence (B, horizon=16, action_dim)

从纯噪声开始，经过 100 步迭代去噪，每步都用 U-Net 预测噪声并用 DDPM 调度器更新。最终得到干净的动作序列。

3.4 FiLM 条件注入

U-Net 如何感知观测信息？通过 FiLM (Feature-wise Linear Modulation)：

FiLM(x) = γ(c) ⊙ x + β(c)   其中 c = [image_features, robot_state]

将观测编码为条件向量，生成逐通道的缩放因子 γ 和偏移 β，注入 U-Net 的每一层卷积。这使得去噪过程条件化于当前观测。

  • 对接触丰富任务表现更好

7. 踩坑与经验

v7 评估 0% 成功率的根因

问题：ACT v7 训练 50K 步 loss 降到 0.1，但 MuJoCo eval 成功率 0%。

根因：数据集 PhangHongHao/so101_push_sim 采集时场景里没有方块！模型只学到了关节运动模式，完全没学到物体交互。

修复：v8 使用 scene_push_table.xml，场景包含台面 + cube + target，重新采集数据训练。

GPU 隔离

✗ --policy.device=cuda:6 数据仍加载到 GPU 0

✓ CUDA_VISIBLE_DEVICES=6 --policy.device=cuda

AMP 语法

✗ --use_amp=true（顶层参数被拒绝）

✓ --policy.use_amp=True（policy-level draccus）

Batch Size 限制

ACT: chunk_size=100 → decoder attention O(n²)

batch_size=128 已经是 85GB 的上限

batch_size=512 直接 OOM (79GB+)

AMP 在 RTX 6000D 上无效

Ada 架构 FP32 已经很快

v3 (无AMP): 3.32s/step

v6 (有AMP): 3.31s/step

几乎无差别
